# 信息化项目验收文档编写

## Overview

本技能专为信息化工程项目验收阶段的文档编写而设计，提供标准化的文档模板和编写指导。支持多种项目文档类型，包括标准验收文档和先建后批场景下的补充文档。

**核心能力：**
- 标准验收文档编写（概要设计、详细设计、产品手册、测试报告等）
- 先建后批场景补充文档编写（可研报告、立项报告）
- 提供文档结构框架和内容要点
- 支持基于项目设计文件的文档生成

## Workflow Decision Tree

根据项目阶段和文档需求选择相应的工作流程：

```
用户请求 → 项目阶段判断
    ├─ 验收阶段文档
    │   ├─ 设计文档类 → 选择：概要设计 / 详细设计 / 架构设计 / 数据库设计
    │   ├─ 产品文档类 → 选择：产品手册
    │   ├─ 测试文档类 → 选择：测试报告
    │   └─ 运维文档类 → 选择：运维报告
    └─ 先建后批补充
        └─ 补充文档类 → 选择：可研报告 / 立项报告
```

## 文档编写流程

### 1. 项目信息收集

在开始编写任何文档前，首先收集以下项目信息：

**基础信息：**
- 项目名称、项目编号
- 建设单位、承建单位
- 项目周期、投资规模
- 项目背景、建设目标

**技术信息：**
- 系统架构设计
- 技术栈选择
- 数据库设计
- 接口设计
- 安全设计

**业务信息：**
- 业务需求
- 功能需求
- 性能需求
- 用户角色

### 2. 文档类型选择

根据用户需求选择对应的文档类型：

#### 标准验收文档

**项目概要设计**
- 适用场景：项目总体设计说明
- 核心内容：总体架构、技术路线、关键设计
- 参考模板：见 `references/概要设计模板.md`

**项目详细设计**
- 适用场景：模块级详细设计说明
- 核心内容：功能模块设计、接口设计、数据结构
- 参考模板：见 `references/详细设计模板.md`

**项目产品手册**
- 适用场景：用户操作指南
- 核心内容：系统功能、操作流程、常见问题
- 参考模板：见 `references/产品手册模板.md`

**测试报告**
- 适用场景：系统测试结果总结
- 核心内容：测试环境、测试用例、测试结果、问题分析
- 参考模板：见 `references/测试报告模板.md`

**项目架构设计报告**
- 适用场景：技术架构详细说明
- 核心内容：架构图、技术选型、部署方案
- 参考模板：见 `references/架构设计模板.md`

**项目数据库设计报告**
- 适用场景：数据库设计详细说明
- 核心内容：ER图、表结构设计、索引设计
- 参考模板：见 `references/数据库设计模板.md`

**项目运维报告**
- 适用场景：运维体系说明
- 核心内容：运维架构、监控方案、应急预案
- 参考模板：见 `references/运维报告模板.md`

#### 先建后批补充文档

**项目可研报告**
- 适用场景：已建设项目补充可行性研究
- 核心内容：项目背景、建设必要性、技术可行性、经济效益
- 参考模板：见 `references/可研报告模板.md`

**项目立项报告**
- 适用场景：已建设项目补充立项申请
- 核心内容：项目概述、建设内容、投资估算、实施计划
- 参考模板：见 `references/立项报告模板.md`

### 3. 文档编写步骤

对于每类文档，遵循以下步骤：

1. **选择文档类型** - 根据需求确定具体文档
2. **收集项目资料** - 获取相关设计文件、技术文档
3. **套用模板框架** - 使用对应模板的结构
4. **填充具体内容** - 根据项目实际情况填写
5. **审核完善** - 检查完整性、准确性

## 使用示例

### 示例1：编写项目概要设计

**用户请求：**
"我需要为XX信息化项目编写概要设计文档，项目是基于微服务架构的业务系统。"

**处理流程：**
1. 确认项目类型：信息化项目，微服务架构
2. 选择文档类型：项目概要设计
3. 收集信息：项目背景、建设目标、技术架构
4. 使用 `references/概要设计模板.md` 作为框架
5. 填写具体内容：
   - 项目概述
   - 总体架构设计
   - 技术路线
   - 关键设计点
   - 实施计划

### 示例2：先建后批场景补充可研报告

**用户请求：**
"系统已经建设完成，现在需要补充项目可研报告用于审批。"

**处理流程：**
1. 确认场景：先建后批，补充文档
2. 选择文档类型：项目可研报告
3. 收集现有资料：系统功能、运行情况、用户反馈
4. 使用 `references/可研报告模板.md` 作为框架
5. 重点补充：
   - 建设必要性（基于实际运行效果）
   - 技术可行性（基于实际技术实现）
   - 经济效益分析（基于实际运行数据）

### 示例3：编写测试报告

**用户请求：**
"系统测试已完成，需要编写完整的测试报告。"

**处理流程：**
1. 选择文档类型：测试报告
2. 收集测试数据：测试用例、执行结果、问题记录
3. 使用 `references/测试报告模板.md` 作为框架
4. 填写内容：
   - 测试环境
   - 测试范围
   - 测试用例执行情况
   - 问题分析与解决
   - 测试结论

## 文档质量要求

### 内容要求
- **完整性**：涵盖所有必要章节和内容点
- **准确性**：技术描述准确，数据真实可靠
- **一致性**：术语使用统一，格式规范
- **可读性**：逻辑清晰，语言简洁专业

### 格式要求
- **标题层级**：使用规范的标题层级（#、##、###）
- **图表使用**：重要设计点配架构图、流程图
- **代码规范**：技术文档中的代码示例要规范
- **引用标注**：引用外部资料需标注来源

### 审核要点
- **技术审核**：技术方案的可行性、先进性
- **业务审核**：业务需求的满足程度
- **规范审核**：是否符合行业标准和规范

## 工作流程建议

### 对于标准验收文档
1. **准备阶段**：收集所有相关设计文件和技术文档
2. **选择模板**：根据文档类型选择对应模板
3. **框架填充**：先完成文档结构框架
4. **内容编写**：逐章节填充具体内容
5. **图表制作**：补充必要的架构图、流程图
6. **审核完善**：多轮审核，确保质量

### 对于先建后批补充文档
1. **现状梳理**：整理系统当前运行情况
2. **反向推导**：基于现状反推可研/立项要素
3. **效益分析**：基于实际运行数据进行效益分析
4. **风险评估**：识别当前系统存在的风险
5. **优化建议**：提出后续优化改进方向

## 注意事项

1. **保密要求**：项目文档可能包含敏感信息，注意保密
2. **版本管理**：文档应有明确的版本号和修订记录
3. **模板适配**：根据项目实际情况调整模板内容
4. **标准遵循**：参考国家相关标准和行业规范
5. **持续更新**：项目变更时及时更新相关文档
